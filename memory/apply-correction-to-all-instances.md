After the user corrected me for under-delivering (choosing the easy option over real user benefit), I
then made the SAME kind of call on the very next decision — deferring a capability on thin "premature"
grounds — which the user had to catch a second time. The correction was applied locally (to the case I
was called out on) but never generalized to the class.

**Why:** a bias is a pattern of reasoning, not a one-off. If you only patch the instance you were
called out on, the same reasoning re-fires on the next decision, and the user pays the cost of
correcting you twice for one underlying flaw.

**How to apply:** treat a correction as a CLASS, not an incident. The moment the user corrects a bias
or a wrong default, re-scan the rest of the current plan / recommendation for other instances of the
SAME reasoning and fix them pre-emptively — before being corrected again.
